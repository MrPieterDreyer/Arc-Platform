    ```markdown
    # ADR-0009 — Customer Authentication Strategy

    **Status:** Open (spike required before Phase 2)
    **Date:** 2026-05-28
    **Deciders:** Pieter Dreyer

    ## Context

    The WC Store API customer endpoint (`GET /customer`) is session-scoped only — it returns
    billing/shipping address data for the current anonymous/authenticated session. It cannot:
    - Look up a customer by ID
    - Return order history across sessions
    - Authenticate (login/logout)

    Full customer functionality requires one of:
    1. **WPGraphQL + JWT** (WPGraphQL JWT Authentication plugin) — GraphQL queries with `Authorization: Bearer` header
    2. **WC REST API + Application Passwords** — REST v3 with HTTP Basic auth (WP 5.6+)
    3. **Custom auth bridge** — WP session cookies passed through to Next.js middleware

    ## Decision

    **Not decided.** A spike is required before Phase 2 (`@arc/next`) to evaluate:
    - JWT flow: install `wp-graphql-jwt-authentication` plugin, test token refresh lifecycle
    - Application Passwords: evaluate security model for headless consumers
    - Cookie bridge: assess SameSite=None cookie propagation complexity

    For Phase 1 (`@arc/core`): `getCustomer()` covers the session-scoped surface.
    `getCustomerOrders()` accepts a `GraphQLClient` configured with `authToken` — the token
    source is deferred to Phase 2.

    ## Consequences

    `useCustomer()` hook (Phase 1) will accept an optional `authToken` parameter.
    `@arc/next` Phase 2 will implement the chosen auth flow and inject the token.
    Until decided: document the authToken parameter clearly in every JSDoc.
    ```
