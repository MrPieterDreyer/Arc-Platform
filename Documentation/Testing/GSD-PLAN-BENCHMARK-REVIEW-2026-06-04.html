ADRs compliant
  • No P0 ship blockers. The core engineering is genuinely strong: @arc/core is provably framework-agnostic (no next/* leakage), caching uses Next 16 'use cache'/cacheTag throughout (zero unstable_cache), and HMAC webhook verification is textbook (timing-safe, dual-timestamp replay window, byte-parity with the PHP signer).
  • The biggest real risk is CI honesty, not code. The visual-regression gate is marked green but has no Linux baselines — it runs on ubuntu-latest and cannot actually pass; and Store API getOrder's positive path is a permanent test.skip with no seeded order. Two "false greens" that overstate coverage.
  • Two process contracts have drifted from their ADRs: the license allowlist (check-licenses.mjs) permits three SPDX IDs not in ADR-0003's locked table, and the v1-publish guard runs only on PRs, not in release.yml — contradicting ADR-0001's own text.
  • Weave SDK + WP plugin are solid and honest. WEAVE-SDK-04/05/09 and WEAVE-NEXT-01 verify in code with matching tests (the 15-input matrix is exact; per-section error isolation degrades to null in prod). Known v0.1 deferrals (WEAVE-NEXT-04, WEAVE-WP-07) are correctly not claimed as shipped.
  • Benchmark verdict: Arc reaches strong parity on the developer/infrastructure layer (cart persistence, caching, preview, section SDK, E2E). The migration-blocking gaps are customer identity (login / orders / addresses — ADR-0009 is still Open) and breadth (Weave ships an SDK, not LOFT's ~30 sections + global theme settings).
  • Housekeeping: a large uncommitted working tree sits on master and GSD planning artifacts are absent (.documentation/.planning/ is empty), so requirement traceability currently rests entirely on E2E-PRIORITIES.md.

Top 5 migration blockers (LOFT Pilot → Arc + Weave)

  1. Customer login — LOFT's whole account area depends on the Customer Account API. Arc has no login flow and ADR-0009 is Open. Hard blocker.
  2. Order history — gated behind the unbuilt auth; only works with a manual test JWT today.
  3. Address book + profile editing — same root cause; Arc reads only session billing.
  4. Section library + global theme settings — LOFT runs ~30 Weaverse sections and 11 setting groups; Weave ships the SDK but not the sections/global-settings UI, so content migration is a rebuild.
  5. Full cart UI — core supports update/remove but the storefront wires only "add," no drawer; LOFT's slide-out + qty steppers + remove would need rebuilding.

Top 5 where Arc matches or exceeds the Pilot baseline

  1. Per-section error isolation — more clearly specified and tested than Weaverse's opaque handling.
  2. E2E coverage + CI matrix — meets or beats LOFT's documented testing.
  3. Cache-invalidation transparency — explicit tag taxonomy + signed webhook gives merchants control the SaaS hides.
  4. Cart-persistence design — the Cart-Token ADR is rigorous (security, ITP, cross-origin, lifecycle).
  5. Draft preview — clean native-Next draftMode() implementation on par with Weaverse.

Intentional deferrals (documented, not bugs)

  • Customer auth / orders / addresses — WC Store API /customer is session-scoped; cross-session auth needs the JWT/app-password bridge ADR-0009 leaves Open. Platform difference + unfinished decision.
  • Section library + 11 global theme-setting groups — Arc ships the SDK for v0.1; Pilot/Templates fill in sections later.
  • First-class Money/Image components — deliberately documented as patterns, not shipped (Stack §5).
  • Hosted payment sandbox — deferred to WC-native gateway handoff in the Pilot starter (PILOT-03).
  • WEAVE-NEXT-04 (cache invalidation on save) & WEAVE-WP-07 (admin UI) — tracked as Phase 4b "when built"; transport plumbing exists and is contract-tested but not claimed as finished. Correctly not scored as regressions.

What Arc builds that Hydrogen doesn't need

Cart-Token cookie layer (Shopify hands you a cart id; WC has no anonymous session); hand-typed Store API + live contract tests (no GraphQL schema); a full PHP WordPress plugin (CPT + REST + validator + webhook + admin app — Weaverse is hosted SaaS); wp-env backend harness + signed revalidate webhook; native-checkout handoff plumbing. Read correctly, these are sound platform adaptations, not deficiencies.
