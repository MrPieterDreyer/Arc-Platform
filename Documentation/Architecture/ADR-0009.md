# ADR-0009 — Customer Authentication Strategy

**Status:** Accepted (2026-06-04 — spike complete)
**Date:** 2026-05-28 (decided 2026-06-04)
**Deciders:** Pieter Dreyer

## Context

The WC Store API customer endpoint (`GET /customer`) is session-scoped only — it returns
billing/shipping address data for the current anonymous/authenticated session. It cannot:
- Look up a customer by ID
- Return order history across sessions
- Authenticate (login/logout)

Full customer functionality requires one of:
1. **WPGraphQL + JWT** (WPGraphQL JWT Authentication plugin) — GraphQL queries with `Authorization: Bearer` header
2. **WC REST API + Application Passwords** — REST v3 with HTTP Basic auth (WP 5.6+)
3. **Custom auth bridge** — WP session cookies passed through to Next.js middleware

## Decision

**Accepted (2026-06-04):** Option 1 — **WPGraphQL JWT**, with the token stored and
transported via the existing **ADR-0006 HttpOnly cookie bridge**.

The key insight from the spike is that this is *not* "JWT vs. cookie bridge": JWT is the
credential **format**, the cookie bridge is the **storage/transport**. They combine:

- **Login** issues a JWT via the `wp-graphql-jwt-authentication` plugin (`login` mutation
  → short-lived access token + long-lived refresh token).
- Tokens are written to two HttpOnly, `SameSite=None; Secure` cookies reusing
  `CART_COOKIE_OPTIONS` from `Arc/Next/src/cookies.ts`: `arc_auth_token` (access) and
  `arc_refresh_token` (refresh).
- `@arc/core`'s `createWPGraphQLClient` already injects `Authorization: Bearer {token}`;
  authenticated queries (`customer { orders, billing, shipping }`) are the cross-session
  source the Store API cannot reach.
- Tokens are **RSC-readable** via `cookies().get()`, identical to `readCartTokenValue()`,
  making auth state available to Server Components — outcome-equivalent to LOFT's Shopify
  `context.customerAccount.login()`.

### Rejected alternatives

- **WC REST + Application Passwords** — a long-lived, non-rotating permanent credential
  with no refresh, and it forces the (excluded) WC core REST API onto catalog queries.
  Retained only as an optional server-to-server adapter, never the customer login path.
- **WP session-cookie bridge** — no WP login session exists for an anonymous, cross-origin
  Next.js visitor; the same reason ADR-0006 rejected nonce auth.

### Token lifecycle

Login → set both cookies → RSC reads the access token → on expiry, `refreshJwtAuthToken`
mints a new access token and retries once (mirroring the nonce-retry-and-rotate loop in
`WooClient.request()`) → logout clears both cookies.

## Consequences

- **`@arc/core`** gains `src/graphql/auth.ts` (login / refresh / logout mutations). The
  existing `getCustomerOrders()` + `createWPGraphQLClient({ authToken })` seam is the
  integration point — no breaking change to the Phase 1 surface. `useCustomer()` keeps its
  optional `authToken` parameter.
- **`@arc/next`** gains `auth-cookies.ts` (read/write the two auth cookies, reusing
  `CART_COOKIE_OPTIONS`), `auth-actions.ts` (login/logout Server Actions), and two new
  cookie-name constants; `createArcClient` wires `authToken` from the access cookie.
- **Dependency / security:** pin `wp-graphql-jwt-authentication` and watch advisories
  (CLAUDE.md notes the WPGraphQL CVE history + small bus factor). The JWT signing secret
  lives in `wp-config.php`, never in the repo. Short access-token TTL bounds blast radius;
  HttpOnly blocks XSS exfiltration.
- **Unblocks the customer track:** login/logout, order history, and address/profile
  editing (benchmark items 4.2 / 4.3).
- Full options analysis and lifecycle detail: see the 2026-06-04 customer-auth spike
  (retained with the audit artifacts as `ADR-0009-SPIKE-2026-06-04.html`).
