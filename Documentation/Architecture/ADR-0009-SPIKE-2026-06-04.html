ADR-0009 — Customer Authentication for Headless WooCommerce

Date: 2026-06-04  ·  Author: Backend Architect spike  ·  For: Pieter Dreyer  ·  Resolves: ADR-0009 (Open → recommend Accepted)

Scope: customer login / logout, order history, address & profile management for a headless Next.js 16 + WC Store API v1 storefront.

Executive Recommendation

Adopt Option 1 — WPGraphQL JWT — but store the credential in HttpOnly cookies via the existing ADR-0006 cookie-bridge. This is not a choice between "JWT" and "cookie bridge": JWT is the credential format, the cookie bridge is the transport & storage. Combining them is the only option that satisfies every constraint and matches the direction the code already took.

Recommended: WPGraphQL JWT + HttpOnly cookie bridge (refresh-token rotation)

1. Problem & Current State

The WC Store API customer surface is session-scoped only. getCustomer() derives billing/shipping from GET /cart (id is 0 for guests), and getOrder() can only return an order that belongs to the current Cart-Token session. There is no login, no cross-session lookup, no order list. Arc/Core/src/store-api/customer.ts:28-41, Arc/Core/src/store-api/orders.ts:1-44, Arc/Core/src/types/customer.ts:1-7

Phase 1 (@arc/core) deliberately shipped the authenticated half as a token-shaped hole: getCustomerOrders(client) takes a GraphQLClient whose authToken callback returns a JWT, and the contract test feeds it from TEST_JWT_TOKEN. Nothing in the repo today issues, stores, or refreshes that token — that is precisely what ADR-0009 must close. Arc/Core/src/__tests__/contract/customer.contract.test.ts:22-25, 73-82, Arc/Core/src/hooks/useCustomer.ts:31-65

Rejected alternatives — why

Option 2 — Application Passwords over HTTP Basic

Application Passwords are a long-lived, effectively-permanent credential designed for machine-to-machine integrations and admin tooling, not end-user storefront login. There is no self-service "create my app password" flow a shopper can perform, no rotation, and a leak exposes a credential with the user's full capability set until manually revoked. It also forces the catalog onto WC core REST — explicitly listed under "what NOT to use" in CLAUDE.md (over-fetched, untyped, N+1 variations). Keep it as an optional server-to-server adapter, never the customer login path.

Option 3 — Pure WP session-cookie bridge

WordPress auth cookies are bound to the WP origin and the nonce/login-session model. For an anonymous Next.js visitor on a different origin (Arc's first-class deployment, per ADR-0006), there is no WP login session to bridge, the WP HttpOnly cookie isn't readable as auth state by the Next origin, and faking it requires brittle reverse-proxy / same-site glue that defeats the cross-origin design. ADR-0006 already rejected nonce-centric auth for the cart for the same reason. High custom-code risk, no upside over JWT.

3. Recommended Design — Token Lifecycle

Two cookies, both set on the Next.js origin by Server Actions / Route Handlers, both following ADR-0006 attribute rules. A short-lived access JWT is sent as Authorization: Bearer to WPGraphQL; a long-lived refresh token mints new access tokens without re-login.

Access cookie
arc_auth_token — JWT, Max-Age ≈ access-token TTL (e.g. 10–15 min), HttpOnly, SameSite=None, Secure, Path /
Refresh cookie
arc_refresh_token — refresh JWT, Max-Age ≈ session window (e.g. 14–30 days), HttpOnly, SameSite=None, Secure, Path /
Attribute source

4. Security Considerations

  • Credential never touches JavaScript. Both cookies are HttpOnly — an XSS payload cannot read or exfiltrate the JWT, identical to the ADR-0006 cart-token rationale. ADR-0006 §Decision, Consequences; Arc/Next/src/cookies.ts:17-23
  • Short access TTL bounds the blast radius. A leaked access token (e.g. via an intercepted bearer header on a misconfigured proxy) expires in minutes; the long-lived secret (refresh token) is only ever sent to the Next origin, never to WPGraphQL in a header.
  • CSRF. SameSite=None is mandatory for cross-origin (ADR-0006), so do not rely on SameSite alone for CSRF. Mutations go through Next.js Server Actions, which already enforce same-origin POST + the framework's action-id protection; add an explicit Origin/Host allow-list check in the login/refresh handlers.
  • Refresh-token rotation + revocation on logout. Rotate the refresh cookie on every use and clear both cookies on logout; expose optional server-side revocation so "log out everywhere" is possible.
  • HTTPS everywhere. SameSite=None ⇒ Secure ⇒ TLS in all envs including local dev (mkcert), already a documented Arc prerequisite; the same ARC_*_COOKIE_SECURE=false override exists for localhost E2E. ADR-0006 §Local development; Arc/Next/src/cookies.ts:8-15
  • Plugin supply-chain. CLAUDE.md flags a recent WPGraphQL core CVE (CVE-2026-33290) and the one-maintainer shape of WPGraphQL-for-WC. Pin the JWT plugin minor, watch advisories, and document a REST fallback for any query GraphQL cannot serve. The JWT secret lives only in wp-config.php (GRAPHQL_JWT_AUTH_SECRET_KEY) — never in the Arc repo (.env.example only).
  • Reserved cookie names. Add arc_auth_token and arc_refresh_token to the reserved list alongside arc_cart_token — never reused for any other purpose. ADR-0006 §Consequences (Neutral)

6. ADR-0009 — Proposed Resolution

Status: Open → Accepted. Decision: customer authentication uses WPGraphQL JWT (via wp-graphql-jwt-authentication), with the access + refresh tokens stored in HttpOnly cookies on the Next.js origin per the ADR-0006 cookie-bridge pattern. Application Passwords (Option 2) and the pure WP-session-cookie bridge (Option 3) are rejected for customer login; Application Passwords may remain an optional server-to-server adapter. The authToken parameter already threaded through @arc/core in Phase 1 is the integration seam — Phase 2 fills it from arc_auth_token.
