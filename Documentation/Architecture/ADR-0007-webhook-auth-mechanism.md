# ADR-0007: WP → Next Revalidation Webhook Auth Mechanism

**Status:** Accepted
**Date:** 2026-05-28
**Deciders:** Arc Platform maintainers
**Phase:** Phase 0
**Related requirements:** TOOL-09 (prep for WEAVE-WP-06, ARC-NEXT-05, WEAVE-NEXT-04)

## Context

When the Weave WP plugin saves a `weave_page` post, or when WooCommerce fires a product update hook, the WP side needs to notify the Next.js frontend to purge its data cache for the affected resource. This is implemented as an HTTP POST from the WP plugin to a Next.js API route (the revalidate handler). The Next.js handler calls `revalidateTag(tag)` to purge the correct cache entry (per ADR-0004 taxonomy).

The revalidate endpoint is publicly reachable on the internet (it must be reachable from the WP server). Without authentication, any party can POST to it and trigger arbitrary cache purges, causing a Denial-of-Cache attack (repeated purges prevent effective caching, degrading site performance).

Authentication requirements:
- The WP server can hold a shared secret (in `wp-config.php`)
- The Next.js server can hold the same secret (in an environment variable)
- No third-party auth service should be required
- The mechanism must be implementable in PHP (WP plugin side) and Node.js (Next.js side) with only stdlib dependencies

## Decision Drivers

- No shared mutable state (avoid token rotation complexity)
- Time-resistant to replay attacks (a captured request cannot be replayed after the window)
- Simple to implement on both PHP and Node.js using only stdlib — no third-party auth library
- Works over HTTPS (the only transport permitted)
- No dependency on WordPress nonces (which require a logged-in WP session) or WP REST authentication

## Considered Options

1. **Static bearer token** — simple; but a captured token is valid forever; no replay protection; secret must be rotated on every deployment; no body integrity guarantee
2. **HMAC body signature (chosen)** — HMAC-SHA256 of the canonical request body; shared secret never transmitted; replay-protected via timestamp; body-integrity guaranteed; implementable in PHP `hash_hmac()` + Node.js `crypto.createHmac()`
3. **Mutual TLS (mTLS)** — strongest authentication; but requires certificate management infrastructure; significant operational complexity for a WP plugin; no realistic path for a community-installable plugin

## Decision

Webhook requests from the WP plugin to the Next.js revalidate handler are authenticated using **HMAC-SHA256** of the raw request body.

**Shared secret:**
- WP side: `WEAVE_WEBHOOK_SECRET` constant defined in `wp-config.php` (e.g. `define('WEAVE_WEBHOOK_SECRET', 'your-secret-here');`)
- Next.js side: `process.env.WEAVE_WEBHOOK_SECRET`
- The secret MUST be at least 32 bytes of cryptographically random data (use `openssl rand -hex 32` to generate)

**Request structure:**

```
POST /api/revalidate HTTP/1.1
Content-Type: application/json
X-Weave-Signature: sha256=<hex-encoded HMAC-SHA256 of raw body>
X-Weave-Timestamp: 2026-05-28T14:00:00Z

{
  "event": "page.updated",
  "tag": "weave:page:home",
  "timestamp": "2026-05-28T14:00:00Z"
}
```

**Headers:**

| Header | Value | Description |
|--------|-------|-------------|
| `X-Weave-Signature` | `sha256=<hex>` | HMAC-SHA256 of the raw UTF-8 body using the shared secret |
| `X-Weave-Timestamp` | ISO8601 UTC | Must match the `timestamp` field in the body; used for replay detection |

**Body fields:**

| Field | Type | Description |
|-------|------|-------------|
| `event` | `string` | Event type: `"page.updated"`, `"product.updated"`, `"collection.updated"` |
| `tag` | `string` | Cache tag to revalidate (from ADR-0004 taxonomy) |
| `timestamp` | `string` (ISO8601) | UTC timestamp of the event; generated by the WP plugin |

**Verification algorithm (Next.js side):**

1. Extract `X-Weave-Signature` and `X-Weave-Timestamp` headers
2. Read the raw request body as a UTF-8 string (before any JSON parsing)
3. Compute `expectedSig = 'sha256=' + HMAC-SHA256(secret, rawBody).hexDigest()`
4. Compare `expectedSig` to `X-Weave-Signature` using a timing-safe comparison (`crypto.timingSafeEqual`)
5. Parse body as JSON; check `body.timestamp` is within ±5 minutes of server UTC time (replay protection window)
6. Validate `body.tag` starts with a known namespace prefix (`arc:` or `weave:`)
7. Call `revalidateTag(body.tag)` and return `200 OK`

**PHP signing algorithm (WP plugin side):**

```php
$body = json_encode($payload, JSON_UNESCAPED_SLASHES | JSON_UNESCAPED_UNICODE);
$sig  = 'sha256=' . hash_hmac('sha256', $body, WEAVE_WEBHOOK_SECRET);
wp_remote_post($next_revalidate_url, [
    'headers' => [
        'Content-Type'       => 'application/json',
        'X-Weave-Signature'  => $sig,
        'X-Weave-Timestamp'  => gmdate('c'),
    ],
    'body'    => $body,
]);
```

**Replay window:** 5 minutes. Requests older than 5 minutes are rejected with `401 Unauthorized`. This window accommodates clock skew between the WP server and the Next.js server.

## Consequences

### Positive

- No mutable token to rotate: the shared secret stays constant; only one secret per deployment pair (WP instance ↔ Next.js instance)
- Replay-protected: a captured request cannot be replayed after 5 minutes; the HMAC covers the body including the timestamp, so altering the timestamp invalidates the signature
- Body integrity: HMAC covers the raw body — a man-in-the-middle cannot alter the `tag` field without breaking the signature
- Stdlib only: PHP `hash_hmac()` + Node.js `crypto.createHmac()` are available everywhere; no third-party auth library needed

### Negative

- Shared secret must be provisioned to BOTH the WP server and the Next.js deployment environment simultaneously — deployment scripts must set both
- Secret rotation requires updating both sides in a coordinated deployment; downtime is not required but there is a brief window (between deployments) where signatures will fail — document mitigation (deploy Next.js side first, then WP side)
- The 5-minute replay window requires both servers to have reasonably accurate clocks (NTP-synced); a WP server with a severely wrong clock will have all webhooks rejected

### Neutral

- The `X-Weave-Signature` header format mirrors GitHub's webhook signature format (`sha256=<hex>`) — familiar to developers who have worked with GitHub webhooks
- Secret generation is not automated by the plugin installer — the merchant or developer must generate and configure it; the Pilot deployment guide documents this step

## Implementation Notes

- Phase 4a (Weave WP plugin): `class-weave-revalidate.php` fires on `save_post_weave_page` and WC product/category hooks; signs and POSTs to the `WEAVE_REVALIDATE_URL` constant from `wp-config.php`
- Phase 2 (`@arc/next`): `createRevalidateHandler({ secret: string, allowedTagPrefixes?: string[] })` factory returns a Next.js route handler (`app/api/revalidate/route.ts`). Default `allowedTagPrefixes: ['weave:', 'arc:']`. Rejects requests with unknown prefixes before calling `revalidateTag`.
- The `WEAVE_WEBHOOK_SECRET` environment variable name is exported as a constant from `@arc/next` (`export const WEAVE_WEBHOOK_SECRET_ENV = 'WEAVE_WEBHOOK_SECRET'`) so it is referenced by name, not by string literal, in both the handler factory and the Pilot starter.
- Pilot starter (Phase 5): `app/api/revalidate/route.ts` uses `createRevalidateHandler` with the env var; `README.md` documents secret generation with `openssl rand -hex 32` and deployment configuration.
