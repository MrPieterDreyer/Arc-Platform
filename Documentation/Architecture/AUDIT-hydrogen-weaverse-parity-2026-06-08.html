Executive summary

Arc/Weave is further along than a v0.1 framework usually is, and in several areas it is ahead of Hydrogen: a more structured typed-error model, runtime Zod validation of Store-API responses, HMAC-signed webhook revalidation with replay protection, a single REST permission choke-point in the WP plugin, and a clean Next.js 16 'use cache' model with zero unstable_cache usage. The supply-chain and release governance baseline (license-check CI, frozen lockfiles, Changesets, SECURITY.md, pnpm catalogs) is strong.

The gaps cluster in three places: (1) security hardening that Hydrogen ships and Arc does not (no CSP / security headers, customer auth designed-but-unbuilt), (2) commerce-storefront primitives and conventions (<Money>/<Image>, error boundaries, SEO, pagination, optimistic line injection), and (3) release-integrity and Phase-2 SaaS isolation (npm provenance, SHA-pinned actions, WPGraphQL version floor; and a multi-tenant model that is entirely greenfield).

The two things to fix before anything else

1. Customer auth is accepted but unbuilt. ADR-0009 specifies WPGraphQL-JWT in HttpOnly cookies, but graphql/auth.ts, auth-cookies.ts, and auth-actions.ts do not exist — order history currently reads a raw process.env.TEST_JWT_TOKEN (load-account-orders.ts:15). There is no real login, rotation, or logout today.

2. No Content-Security-Policy or security headers anywhere. Hydrogen ships createContentSecurityPolicy() + useNonce(); Arc's example app sets only cacheComponents (minimal-app/next.config.ts:3). For a framework that renders merchant-authored page-config and handles carts, the absence of CSP, X-Frame-Options, nosniff, HSTS, and Referrer-Policy is the single largest parity gap.

Where Arc/Weave is already ahead of the reference

  • Typed discriminated-union error model (Arc/Core/src/types/errors.ts:34) vs Hydrogen's flat GraphQL error array.
  • Runtime Zod validation of cart responses, warn-not-throw (store-api/validate.ts:42).
  • HMAC-SHA256 webhook revalidation with ±5-min replay window + tag allowlist (Arc/Next/src/revalidate.ts:52) — Hydrogen delegates purge to Oxygen.
  • Single require_cap() REST choke-point with an explicit 401/403 auth-matrix test (class-weave-rest-controller.php:118, test-rest-auth.php).
  • Server-minted section IDs + 1 MB body gate + strict unknown-key rejection in the WP plugin — stricter than Weaverse's client-trusting model.
  • Deploy-target agnostic (pure Next.js) where Hydrogen is coupled to Oxygen.

Remediation roadmap

Sequenced into waves. Effort: S <½ day · M 1–3 days · L >3 days. Every Partial/Missing matrix row maps to an item here.

Wave 1 — Security & release blockers

do first
  • Implement customer auth per ADR-0009. L
    Build @arc/core/src/graphql/auth.ts (login/refresh/logout), @arc/next/src/auth-cookies.ts + auth-actions.ts; reuse CART_COOKIE_OPTIONS; reject empty/short tokens; remove TEST_JWT_TOKEN read. Unblocks all account pages. (H-2)
  • Ship CSP + security headers. M
    New @arc/next/src/security-headers.ts (nonce CSP mirroring Hydrogen, X-Frame-Options, nosniff, HSTS, Referrer-Policy); wire via Pilot proxy.ts + example next.config.ts headers(). Pin WC checkout + gateway origins in connectSrc. (H-1, → ADR-0010)
  • shouldCacheResult guard in catalog loaders. S
    Guard loadProduct/loadProducts/loadCollection/loadCollectionProducts/cachedConfig so null / partial-error responses are not cached for the full TTL. (CACHE-P1)
  • npm provenance + OIDC Trusted Publishing. S
    Append --provenance to changeset publish; register packages for OIDC; set NPM_TOKEN: ''. (S-1, → ADR-0012)
  • SHA-pin all GitHub Actions. S
    Replace mutable tags in ci.yml/release.yml; let the existing github-actions Dependabot manage SHA bumps. (S-2)
  • Declare WPGraphQL version floor. S
    Add Requires Plugins: wp-graphql, wp-graphql-woocommerce to weave.php + readme minimum past CVE-2026-33290. (S-3)

ADR proposals

Three new draft ADRs accompany this report (Markdown, alongside the existing ADR-*.md series). Two existing ADRs need implementation, not redecision:

  • ADR-0010 — Security-header & CSP policy (new, draft) — nonce-based CSP, frame/transport/referrer headers, where they live in @arc/next + Pilot proxy.ts.
  • ADR-0011 — Multi-tenant isolation for Phase-2 SaaS (new, draft) — WP Multisite per tenant, tenant-scoped cache tags, preview sandboxing, per-tenant authz.
  • ADR-0012 — Supply-chain hardening (new, draft) — npm provenance/OIDC, SHA-pinned actions, WPGraphQL version floor, Composer/Dependabot coverage.
  • ADR-0009 (existing, Accepted) — customer auth: build it (Wave 1). No decision change.
  • ADR-0004 (existing, Accepted) — caching: add the shouldCacheResult guard + document the TTL table. No decision change.

Evidence gaps & caveats

Stated explicitly rather than guessed, per the audit's evidence rules.

  • Pilot starter (Arc/Pilot/) has no app/ directory found — error-boundary / SEO patterns were verified only against Arc/Next/examples/minimal-app/.
  • .env.example could not be read (sandbox treats .env* as secret-bearing); .gitignore:23-25 confirms only .env.example is tracked. Recommend a manual eyeball that it holds placeholders only.
  • next.config.ts resolved TTL/SWR values for 'hours'/'minutes' profiles live in the consumer app, not @arc/next — the actual numbers were not confirmed.
  • WP plugin PHPUnit tests beyond test-rest-auth.php were not located; auto_update_plugin hook presence is [INFERENCE] (WP defaults off for non-.org plugins).
  • pnpm-lock.yaml not read — the @tanstack/react-query@5.101.0 exclusion vs caret resolution was not verified against the lockfile.
  • WordPress Multisite compatibility of class-weave-cpt.php is untested/undocumented.
  • Live IDOR enforcement on getOrder/getCustomerOrders depends on WooCommerce/WPGraphQL server-side scoping — asserted from error-code handling, not tested against a live store.
